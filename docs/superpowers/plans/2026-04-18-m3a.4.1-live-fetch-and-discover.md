# M3a.4.1 — Upstream Live Fetch + Discover Module + Pipeline CI Job Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add the live-upstream fetch code paths that every provider ingest is missing today, introduce a `pipeline/discover/` module that diffs upstream version-indicator state and emits a `changed-shards.json` manifest for downstream matrix fan-out, and start actually running the Python pipeline on every PR by adding a `pipeline` job to `.github/workflows/ci.yml`. This is the foundation that `m3a.4.2` (`data-daily.yml` end-to-end) builds on. Nothing in this plan publishes shards or touches the cron schedule — that is strictly `m3a.4.2`.

**Architecture:**

1. **Provider live fetchers** (`pipeline/ingest/{aws,azure,gcp}_common.py` + existing `openrouter` `LiveClient`): one function per provider that downloads the upstream price surface for a given shard into a local file and returns the upstream version indicator (publicationDate / ETag / synthesized hash). Streaming reads, bounded retry, `requests.Session` with `User-Agent: sku-pipeline/<version> (+https://github.com/sofq/sku)`. All callers receive the local path so existing `--offer` / `--prices` / `--skus` flags keep working unchanged.
2. **Discover module** (`pipeline/discover/__init__.py` + per-provider sub-modules): runs once per pipeline invocation, fetches cheap version indicators only (no full offer downloads), loads the previous run's state from `$DISCOVER_STATE_DIR/state.json`, diffs, emits `changed-shards.json` on stdout in a shape `{"shards": ["aws-ec2", "aws-rds", ...], "baseline_rebuild": false}`. `baseline_rebuild` is true when either the workflow hints `--baseline-rebuild` (for the 1st/15th-of-month branch that `m3a.4.2` will wire up) or the state file is missing (first run / cache miss).
3. **Make targets**: `make shard SHARD=<name> LIVE=1 SRC=<path>` — when `LIVE=1` is set, the target skips `FIXTURE=` and wires the downloaded price file in via `--offer` / `--prices` / `--skus`. A new `make discover` target runs the discover module against the packaged state cache (no network by default; `DISCOVER_LIVE=1` enables real network calls, only used in CI).
4. **Pipeline CI job**: a new `pipeline` job in `.github/workflows/ci.yml`, ubuntu-latest + Python 3.11, that runs `make pipeline-test` and `make openrouter-shard aws-shards azure-shards gcp-shards` against the existing fixtures. This proves the Python pipeline actually executes on every PR/push. The live-upstream code paths added in this plan are unit-tested with `requests-mock` — they never hit the real network from CI during this sub-milestone.

**Tech Stack:** Python 3.11+, `requests` (already a pipeline dep), `requests-mock` (new dev dep), existing `pipeline/package/build_shard.py` (unchanged), existing `ingest/` modules (only `argparse` surface + new `_common.py` helpers touched). No Go changes. No new runtime deps.

**Spec ref:** `docs/superpowers/specs/2026-04-18-sku-design.md` §3 *Update flow (daily)* job 1 (discover) + job 2 (ingest fetch shape), §7 (workflow list), §6 *Data pipeline tests*. Fills in the m3a.4 deferral spots from `2026-04-18-m3a.3-dynamodb-cloudfront-updater.md` lines 20–23 (`Raw sibling shards`, `Manifest fetch + ETag + delta chain`, `data-daily.yml`, `Validation harness` — this plan only tackles the fetch + discover half of those deferrals).

**Explicit non-goals for m3a.4.1** (declared up front to prevent scope creep):
- `data-daily.yml` cron workflow → `m3a.4.2`.
- Publishing releases / manifest.json / jsDelivr purge → `m3a.4.2`.
- DuckDB Parquet intermediate + `EXCEPT`-join delta packaging → `m3a.4.2`.
- Full daily baseline rebuild on day-of-month ∈ {1, 15} → `m3a.4.2`.
- `data-validate.yml` weekly upstream re-fetch cross-check → `m3a.4.3`.
- OIDC IAM role / GCP Workload Identity Federation setup → `m3a.4.3`.
- Client-side `internal/updater` delta-chain walking + ETag caching + stable/daily channel split → `m3a.4.3`.
- `data-baseline.yml` as a separate workflow file — spec §3 line 183 folds baseline into `data-daily.yml`'s diff+package job, which is simpler; this plan follows §3 over §7's list. If maintainers later insist on a separate workflow, it's a trivial split and out of scope here.
- Raw sibling shards (`aws-ec2-raw.db.zst`, etc.) — the discover module emits shard IDs for normalized shards only; raw-shard packaging is a `m3a.4.2` concern, and this plan's fetchers intentionally leave the raw offer files in `$DISCOVER_OUT/{provider}/{shard}/offer.json` so `m3a.4.2`'s packager can reuse them without re-downloading.

**Shards covered:** the full 17-shard surface shipped through m3b.4. `aws-ec2`, `aws-rds`, `aws-s3`, `aws-lambda`, `aws-ebs`, `aws-dynamodb`, `aws-cloudfront`, `azure-vm`, `azure-sql`, `azure-blob`, `azure-functions`, `azure-disks`, `gcp-gce`, `gcp-cloud-sql`, `gcp-gcs`, `gcp-run`, `gcp-functions`, plus `openrouter`. ## Task 2: AWS live fetcher (`aws_common.fetch_offer`)

**Files:**
- Modify: `pipeline/ingest/aws_common.py`
- Create: `pipeline/tests/test_aws_fetch.py`

**Contract:**

```python
# pipeline/ingest/aws_common.py

_AWS_OFFER_BASE = "https://pricing.us-east-1.amazonaws.com/offers/v1.0/aws"
_AWS_SERVICE_CODES: dict[str, str] = {
    "aws_ec2":       "AmazonEC2",
    "aws_rds":       "AmazonRDS",
    "aws_s3":        "AmazonS3",
    "aws_lambda":    "AWSLambda",
    "aws_ebs":       "AmazonEC2",           # EBS prices live inside the EC2 offer
    "aws_dynamodb":  "AmazonDynamoDB",
    "aws_cloudfront":"AmazonCloudFront",
}

def fetch_offer(shard: str, target: Path, *, session=None, retries: int = 3) -> str:
    """Download an AWS offer index.json for `shard` into `target`.

    Uses streaming GET to avoid materialising ~300MB payloads in RAM.
    Returns the upstream `publicationDate` string from the JSON (used as the
    version indicator by `pipeline/discover/aws.py`). Raises RuntimeError on
    exhausted retries or truncated responses (Content-Length mismatch).
    """
```

- [ ] **Step 1: Write failing unit tests first (TDD).**

`pipeline/tests/test_aws_fetch.py` covers:
- Happy path: `requests_mock` serves a valid `index.json` for `AmazonEC2`; `fetch_offer("aws_ec2", tmp_path/"offer.json")` returns the offer's `publicationDate` and writes the body byte-identical to the mocked response.
- EBS shares the EC2 offer: `fetch_offer("aws_ebs", ...)` hits `.../AmazonEC2/current/index.json` (verified via `requests_mock.last_request.url`).
- 404 → `RuntimeError` containing the upstream URL (no retry loop for 404).
- 500 on first two attempts, 200 on third → success; `requests_mock.call_count == 3`.
- 500 on all attempts → `RuntimeError`; `requests_mock.call_count == retries`.
- Truncated body (Content-Length claims 1000, body is 500 bytes) → `RuntimeError("truncated")`; target file is not left behind (`not target.exists()`).
- User-Agent header matches `sku-pipeline/` prefix.

Run the tests — they should all fail (function does not exist yet).

- [ ] **Step 2: Implement `fetch_offer`.**

Streaming GET via `session.get(url, stream=True, timeout=60)`. Accumulate chunks into a `.part` tempfile sibling to `target`, verify final length against `Content-Length`, then `os.replace(.part, target)` for atomic publish. Parse the trailing JSON's `publicationDate` via a second read (fine — the file is on local disk now; no need to hold the bytes in RAM). Retry 500/502/503/504 with `time.sleep(0.5 * 2**attempt)` like the existing `LiveClient`. Do not retry 4xx.

**Contract:**

```python
# pipeline/ingest/azure_common.py

_AZURE_RETAIL_BASE = "https://prices.azure.com/api/retail/prices"

def fetch_prices(filter_str: str, target: Path, *, session=None, retries: int = 3) -> str:
    """Page through Azure retail-prices with `$filter=<filter_str>` and write
    the concatenated `Items` array to `target` as a single JSON document
    {"Items": [...]}. Returns the SHA256 hex digest of the sorted, normalised
    item list — used as the version indicator by `pipeline/discover/azure.py`.

    Follows `NextPageLink` until empty; caps at 10_000 pages to prevent runaway.
    Each shard in `pipeline/discover/azure.py` defines its own `filter_str`
    (e.g. `serviceName eq 'Virtual Machines' and armRegionName eq 'eastus'`).
    """
```

- [ ] **Step 1: Write failing unit tests (TDD).**

`pipeline/tests/test_azure_fetch.py` covers:
- Two-page pagination: `requests_mock` returns `{"Items": [...3 items...], "NextPageLink": "<url2>"}` then `{"Items": [...2 items...], "NextPageLink": null}`. `fetch_prices` writes all 5 items to the target; `json.loads(target)` has a single `Items` key; returned hash is stable across runs (sort the items by `skuId` before hashing).
- Empty result: returns hash of `[]`, writes `{"Items": []}`.
- Filter string with apostrophes is URL-encoded in the initial GET (verified via `requests_mock.last_request.query`).
- Runaway guard: 10001 pages → `RuntimeError("page_cap_exceeded")`.
- 500 on page 2, retry succeeds → total request count reflects retries.

- [ ] **Step 2: Implement `fetch_prices`.**

First request: `GET _AZURE_RETAIL_BASE?$filter=<urlencoded filter_str>&api-version=2023-01-01-preview`. Subsequent requests follow `NextPageLink` verbatim (Azure returns full URLs). Accumulate items into a list; after all pages, write `json.dumps({"Items": sorted_items}, separators=(",", ":"), sort_keys=True)` to `target.with_suffix(target.suffix + ".part")`, then `os.replace`. Hash is `hashlib.sha256(json.dumps(sorted_items, ...).encode()).hexdigest()`.

```python
# pipeline/discover/aws.py

def discover(shards: Iterable[str], *, session=None) -> dict[str, str]:
    """Return {shard_id: indicator} for the given shard ids.

    For AWS, the indicator is the offer's `publicationDate` string fetched
    from `.../AmazonEC2/current/index.json` etc. — a cheap HEAD-ish call
```

## Task 8: Discover driver (`pipeline/discover/__main__.py`)

**Files:**
- Create: `pipeline/discover/__main__.py`
- Create: `pipeline/discover/driver.py`  (extracted logic, importable for tests)
- Create: `pipeline/tests/test_discover_driver.py`

**Contract:**

```bash
python -m pipeline.discover \
  --state      path/to/state.json   \
  --out        path/to/changed.json \
  [--live]                          \  # when absent, error out rather than return empty
  [--baseline-rebuild]              \  # force every shard into the output
  [--shards aws-ec2,gcp-gce,...]    \  # optional allowlist; default = all known
  [--gcp-api-key-env GCP_BILLING_API_KEY]
```

**Output schema (`changed.json`):**

```json
{
  "schema_version": 1,
  "baseline_rebuild": false,
  "generated_at": "2026-04-19T03:00:00Z",
  "shards": ["aws-ec2", "gcp-gce"],
  "unchanged": ["aws-rds", "aws-s3", "..."],
  "errors": [
    {"shard": "azure-vm", "reason": "http_500", "detail": "..."}
  ]
}
```

On any shard-level error, the discoverer records it under `errors` and continues. If **all** shards error out, the exit code is 2 (`pipeline_failure`). A non-empty `errors` array with at least one successful shard exits 0 — `m3a.4.2`'s workflow fans out over `shards` and files a GitHub issue for each entry in `errors` (scope of that plan).

- [ ] **Step 1: Unit tests (TDD).** Using fixtures + `requests_mock`:
  - First run (no state file): all shards appear in `shards`, `baseline_rebuild: true`.
  - Second run with unchanged upstream: `shards: []`, `baseline_rebuild: false`.
  - One shard's upstream changed: only that shard in `shards`.
  - `--baseline-rebuild` forces all shards into `shards`, sets flag true.
  - `--shards aws-ec2,aws-rds` restricts the check to those two only (unknown shard id → exit 4 `validation`).
  - One shard's fetch errors: included under `errors`, not in `shards`; others still classified correctly; exit 0.
  - All shards error: exit 2.

- [ ] **Step 1: Append a new `pipeline` job.**

```yaml
  pipeline:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.11"
          cache: pip
          cache-dependency-path: pipeline/pyproject.toml
      - name: Install pipeline deps
        run: |
          python -m pip install --upgrade pip
          python -m pip install -e 'pipeline[dev]'
      - name: Lint pipeline

      - name: Pipeline unit tests
        run: make pipeline-test
      - name: Fixture shard builds (every provider)
        run: make openrouter-shard aws-shards azure-shards gcp-shards
      - name: Dry-run discover against empty state
        run: make discover
      - name: Upload fixture shards as artifact
        uses: actions/upload-artifact@v4
        with:
          name: fixture-shards
          path: dist/pipeline/*.db
          retention-days: 7
```

**Notes on deliberate choices:**
- This job does **not** run `--live` — it is a deterministic, fixture-only CI check. Live ingest lives in `data-daily.yml` (m3a.4.2) where it has a cron schedule.
- The `Lint pipeline` step first tries the venv-installed ruff (matches `pipeline/Makefile`'s `RUFF` var), falling back to a PATH ruff if Make hasn't created the venv (shouldn't happen given the step order, but keeps the job robust to refactors).
- `actions/cache` via `setup-python` keys on `pipeline/pyproject.toml` so version bumps bust the cache.
- The upload-artifact step lets a PR reviewer download the fixture shards and smoke-test them with the Go binary from the matrix job.
