# M3a.4.3 — `data-validate.yml` OIDC Cross-Check + Client `internal/updater` Delta-Chain Walking Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Close the data-release feedback loop by adding (1) a CI-side validation workflow that weekly re-fetches sampled SKUs from the upstream providers via short-lived OIDC federation and fails the release on >1% price drift, plus an offline daily EC2 cross-check against `vantage-sh/ec2instances.info`; and (2) the client-side half of §3's `sku update` flow — delta-chain walking, ETag-cached manifest fetches, and the `stable`/`daily` channel split — so users actually benefit from the daily deltas that `m3a.4.2` publishes. Until this plan lands, `sku update` still downloads baselines only and users burn the full-shard-size every update cycle.

**Architecture:**

1. **`data-validate.yml`** — weekly (Monday 04:00 UTC) + `workflow_dispatch`. One matrix over on-demand shards: per shard, stratified-sample ~20 SKUs from the catalog, re-fetch the same SKU from upstream via per-provider auth (AWS OIDC IAM role → SigV4 `GetProducts`; Azure anonymous filtered query; GCP OIDC via Workload Identity Federation → `services.skus.list`; OpenRouter anonymous), compare `|catalog - upstream| / upstream < 0.01`, and on drift >1% file a `catalog-drift` issue and exit the workflow nonzero. `*-commitments`, `*-reservations`, `*-cud` shards are included here (daily `data-daily.yml` already sanity-checks them via row-count drift — the full lookup only runs weekly, spec §3 line 200). EC2 runs an additional offline cross-check vs a bundled snapshot of `vantage-sh/ec2instances.info/instances.json` downloaded at validate time.
2. **OIDC infrastructure** — maintainer-provisioned, documented in `docs/ops/validation.md`:
   - AWS: IAM role `arn:aws:iam::<acct>:role/sku-validator` with `AWSPriceListServiceFullAccess` (read-only), trust policy scoped to `repo:sofq/sku:ref:refs/heads/main` + `repo:sofq/sku:pull_request` for PR-time dry-runs. Stored in `AWS_VALIDATE_ROLE_ARN` repo variable (not a secret — ARNs are not sensitive).
   - GCP: Workload Identity Pool + provider configured for GitHub OIDC; service account `sku-validator@<project>.iam` with `roles/billing.viewer`. Variable: `GCP_WIF_PROVIDER`, `GCP_VALIDATE_SA`.
   - Azure / OpenRouter: no federation; anonymous HTTPS.
3. **`internal/updater` delta-chain walking** — extend the package m3a.3 created with:
   - `FetchManifest(ctx, channel string) (*Manifest, string /* etag */, error)` with HTTP `If-None-Match` support and 5xx-fallback to the jsDelivr mirror.
   - `Update(ctx, shard string, opts Options) error` — the delta-chain driver: if `local.head_version < remote.head_version` and chain length ≤ `max_delta_chain` (default 20, per spec §3 line 263), walk the chain, download each `.sql.gz`, verify sha256, apply within one SQLite transaction; on chain-too-long or `from`-mismatch (§3 line 265 "force-push race"), fall back to baseline.
   - `Channels` — `stable` (default; baselines only, never picks up same-day deltas) vs `daily` (baselines + deltas; typical agent choice). Configured via `config.yaml` `update.channel: daily` + `--channel` flag + `SKU_UPDATE_CHANNEL` env.
4. **Client-side sha256 + atomic apply** — existing `Install` path already verifies baseline sha256 and does atomic rename; the new `Update` path does the same per delta plus a `BEGIN IMMEDIATE; ... COMMIT;` wrap around the SQL replay, with an explicit rollback on any delta-level failure. No half-applied state on disk.

**Tech Stack:** GitHub Actions, `aws-actions/configure-aws-credentials@v4` (AWS OIDC), `google-github-actions/auth@v2` (GCP WIF), `aws/aws-sdk-go-v2` (new Go dep — SigV4 + PriceList service; trimmed build tag to only pull `service/pricing` to keep binary size flat), existing `modernc.org/sqlite`, `github.com/gofrs/flock` (already referenced in spec §3 line 267 — already vendored per m3a.3), `requests` (Python, already present). No new Python deps.

**Spec ref:** `docs/superpowers/specs/2026-04-18-sku-design.md` §3 *Update flow job 4* (lines 185-203) + §3 *Client `sku update` flow* (lines 260-267) + §3 *Reliability* (line 272 `actions/attest-build-provenance`) + §6 *Validation harness* (lines 953-966) + §7 *Workflows* (line 1024 `data-validate.yml`). Fills in the last m3a.4 deferral listed in `2026-04-18-m3a.3-dynamodb-cloudfront-updater.md` line 21.

**Explicit non-goals for m3a.4.3** (declared up front to prevent scope creep):
- Code-signing shards with cosign or `actions/attest-build-provenance` → `m7.3-security-bench-release`. The shard manifest still lists only sha256 checksums after this plan.
- Property-based delta-chain tests via `pgregory.net/rapid` (spec §6 *Property-based tests*) → `m7.3`.
- Release-alerts email webhook on purge failure (§3 line 213) — `m3a.4.2` already files a GH issue on failure; the email webhook is a nice-to-have and lives in a maintainer's incident-response tooling, not in this plan.
- Reservation-lookup deep SigV4 paths for `*-commitments`/`*-reservations`/`*-cud`: these shards don't exist yet (spec §3 defers them past M3a). The validation harness *structure* accommodates them (matrix entry per shard, sampling function keyed on kind) but no commitment shards are actually validated in this plan.
- `deprecated_shards` flag in the manifest (spec §3 line 278) — not exercised today; revisit when a provider actually objects.
- HTTP retries on the authoritative manifest URL — the `FetchManifest` flow treats any 5xx/timeout on `github.com/sofq/sku/releases/latest/download/manifest.json` as immediate fallback to jsDelivr; no retry loop on the primary. This is spec-aligned (§3 line 224: "fall back to jsDelivr on 5xx / timeout").

**Shards covered (validate):** same 17-shard surface as m3a.4.2. On-demand only (no `*-commitments` etc. today). Sampling is `max(20, rows/50_000)` SKUs per shard per run.

- [ ] **Step 1: Write workflow.**

```yaml
name: data-validate

on:
  schedule:
    - cron: "0 4 * * 1"         # Mondays 04:00 UTC
  workflow_dispatch:
    inputs:
      shards:
        description: "Comma-separated shards to validate (default: all)"
        required: false
        default: ""

permissions:
  contents: read
  id-token: write               # OIDC federation
  issues:   write               # drift issue filing
  actions:  read

concurrency:
  group: data-validate
  cancel-in-progress: true

jobs:
  resolve:
    runs-on: ubuntu-latest
    outputs:
      shards: ${{ steps.resolve.outputs.shards }}
      catalog_version: ${{ steps.resolve.outputs.catalog_version }}
    steps:
      - uses: actions/checkout@v4
      - name: Resolve latest data release + shard list
        id: resolve
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          REQUESTED: ${{ github.event.inputs.shards }}
        run: |
          tag=$(gh release list --limit 1 --json tagName --jq '.[0].tagName')
          mkdir -p dist/pipeline/validate
          gh release download "$tag" --pattern manifest.json --dir dist/pipeline/validate/
          if [ -n "$REQUESTED" ]; then
            shards=$(echo "$REQUESTED" | jq -R 'split(",")')
          else
            shards=$(jq -c '.shards | keys' dist/pipeline/validate/manifest.json)
          fi
          echo "shards=$shards" >> "$GITHUB_OUTPUT"
          echo "catalog_version=${tag#data-}" >> "$GITHUB_OUTPUT"

  validate:
    needs: resolve
    runs-on: ubuntu-latest
    strategy:
      fail-fast: false
      matrix:
        shard: ${{ fromJSON(needs.resolve.outputs.shards) }}
    steps:
      - uses: actions/checkout@v4
      - uses: ./.github/actions/setup-pipeline
      - name: Install validate extras
        run: python -m pip install -e 'pipeline[validate]'
      - name: Configure AWS credentials (OIDC)
        if: startsWith(matrix.shard, 'aws-')
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ vars.AWS_VALIDATE_ROLE_ARN }}
          aws-region: us-east-1
      - name: Authenticate to Google Cloud (WIF)
        if: startsWith(matrix.shard, 'gcp-')
        uses: google-github-actions/auth@v2
        with:
          workload_identity_provider: ${{ vars.GCP_WIF_PROVIDER }}
          service_account: ${{ vars.GCP_VALIDATE_SA }}
      - name: Download shard from latest release
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          CV: ${{ needs.resolve.outputs.catalog_version }}
          SHARD: ${{ matrix.shard }}
        run: |
          gh release download "data-${CV}" --pattern "${SHARD}.db.zst" --dir dist/pipeline/
          zstd -d "dist/pipeline/${SHARD}.db.zst"
      - name: Download vantage instances.json (EC2 only)
        if: matrix.shard == 'aws-ec2'
        run: |
          curl -sSL --fail \
            https://raw.githubusercontent.com/vantage-sh/ec2instances.info/master/www/instances.json \
            -o dist/pipeline/instances.json
      - name: Run validator
        env:
          SHARD: ${{ matrix.shard }}
        run: |
          extra=()
          if [ "$SHARD" = "aws-ec2" ]; then
            extra+=(--vantage-json dist/pipeline/instances.json)
          fi
          python -m pipeline.validate \
            --shard "$SHARD" \
            --shard-db "dist/pipeline/${SHARD}.db" \
            --report "dist/pipeline/${SHARD}.report.json" \
            "${extra[@]}"
      - uses: actions/upload-artifact@v4
        with:
          name: validate-${{ matrix.shard }}
          path: dist/pipeline/*.report.json
      - name: File drift issue on failure
        if: failure()
        env:
          GH_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          SHARD: ${{ matrix.shard }}
          CV: ${{ needs.resolve.outputs.catalog_version }}
        run: |
          gh issue create \
            --title "Catalog drift in ${SHARD} (catalog ${CV})" \
            --label "pipeline,catalog-drift" \
            --body "See workflow run ${GITHUB_SERVER_URL}/${GITHUB_REPOSITORY}/actions/runs/${GITHUB_RUN_ID} and the attached validate-${SHARD} artifact."
```

```go
type Manifest struct {
    SchemaVersion   int                      `json:"schema_version"`
    GeneratedAt     time.Time                `json:"generated_at"`
    CatalogVersion  string                   `json:"catalog_version"`
    Shards          map[string]ShardEntry    `json:"shards"`
}
type ShardEntry struct {
    BaselineVersion    string  `json:"baseline_version"`
    BaselineURL        string  `json:"baseline_url"`
    BaselineSHA256     string  `json:"baseline_sha256"`
    BaselineSize       int64   `json:"baseline_size"`
    HeadVersion        string  `json:"head_version"`
    MinBinaryVersion   string  `json:"min_binary_version"`
    ShardSchemaVersion int     `json:"shard_schema_version"`
    Deltas             []Delta `json:"deltas"`
    RowCount           int64   `json:"row_count"`
    LastUpdated        time.Time `json:"last_updated"`
}
type Delta struct {
    From   string `json:"from"`
    To     string `json:"to"`
    URL    string `json:"url"`
    SHA256 string `json:"sha256"`
    Size   int64  `json:"size"`
}

type ManifestSource interface {
    Fetch(ctx context.Context, etag string) (m *Manifest, newETag string, notModified bool, err error)
}

func NewHTTPSource(primary, fallback string, rt http.RoundTripper) ManifestSource
```

Behaviour of `HTTPSource.Fetch`:
- GET `primary` with `If-None-Match: <etag>`; 304 → `notModified=true`; 200 → parse body, return new `ETag`; 5xx/timeout → retry once on `fallback`; any other error → surface.
- `notModified=true` skips all delta/baseline work — the caller's cached manifest is authoritative for this run.

## Task 10: `internal/updater` delta-chain walker (Go, TDD)

**Files:**
- Create: `internal/updater/chain.go`
- Create: `internal/updater/chain_test.go`
- Create: `internal/updater/testdata/delta-fixtures/...`

**Contract:**

```go
type Applier struct {
    HTTPClient  *http.Client
    DBPath      string        // absolute path to the shard file
    MaxChain    int           // default 20
    OnProgress  func(event string, delta Delta)
}

func (a *Applier) Apply(ctx context.Context, from, to string, chain []Delta) error
```

Algorithm:
1. If `len(chain) == 0` or `from != chain[0].From`: return `ErrChainStartsElsewhere` (caller falls back to baseline).
2. If `len(chain) > MaxChain`: return `ErrChainTooLong`.
3. Open shard via `modernc.org/sqlite`; take advisory flock per spec §3 line 267 (reuse `internal/sqliteutil.Flock` — if no such helper, create one in this plan, tests included).
4. `BEGIN IMMEDIATE;` — all deltas apply in one transaction. If any delta fails mid-stream, `ROLLBACK;` and return the error.
5. For each delta: GET url, verify sha256, gunzip, `tx.ExecContext(sqlBody)`; call `OnProgress("delta-applied", delta)`.
6. `UPDATE metadata SET generated_at=..., catalog_version=...;` — match the manifest's `head_version`.
7. `COMMIT;`.
